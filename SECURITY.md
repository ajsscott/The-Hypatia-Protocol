# Security

## What "security" means here

This is a local-first framework. Markdown files, JSON stores, and Python scripts that run on the Scholar's Mac with her permissions. No API, no authentication layer, no network surface, no database server.

The actual security surface is small:

| Component | What it does | Risk |
|---|---|---|
| **Git filter scripts** | Sanitize sensitive data on commit | A bypass could leak data intended to scrub |
| **Setup script** | Install deps, configure git | Runs with user permissions, no elevation |
| **Vectorstore scripts** | Build / query local embeddings | Local file I/O only, no network |
| **MCP server** | Expose KB search to Roo Code | stdio only, no network listener |
| **Fetch security proxy** | Filter URLs before they reach mcp-server-fetch | Blocks private IPs, metadata endpoints, URL shorteners, dangerous ports |

## Security architecture

Designed with a minimal attack surface. Comparison against threat categories that affect AI agent systems broadly:

### No network exposure

There is no server, no gateway, no daemon, no port binding. The MCP server (`kb_server.py`) communicates via stdio with the local Roo Code extension that spawned it. Nothing is discoverable by network scanners.

### No credential storage

The framework stores zero credentials. No API keys, no OAuth tokens, no bot tokens. Roo Code (the substrate) handles its own LLM-provider authentication independently. Hypatia never sees or touches those credentials.

### No third-party code execution

There is no skill marketplace, no plugin registry, no package manager for extensions. Protocols are markdown files that instruct behavior. Adding a protocol means writing a `.md` file, not running `npm install`. All executable code ships in the repo and is auditable:

**Core (runs during setup or normal operation):**
- `scripts/setup.sh` (one-time at setup)
- `scripts/git-filter-clean.py` (on git commit)
- `scripts/git-filter-smudge.py` (on git checkout)
- `scripts/secure-fetch.py` (as MCP fetch proxy)
- `scripts/harden-repo.sh` (pre-push scan, on-demand)
- `scripts/save-session.py` (atomic save operations)

**Vectorstore (optional, on-demand):**
- `hypatia-kb/vectorstore/kb_server.py`, `kb_query.py`, `kb_sync.py`, `kb_vectorize.py`, `concat.py` (local file I/O only, with hypothesis-based tests)

**Maintenance utilities (manual, on-demand):**
- `scripts/full-maintenance.sh`, `scripts/python-maintenance.sh` (Python cache cleanup; no network calls)

**Schema tools (development, on-demand):**
- `scripts/normalize-schemas.py`, `scripts/validate-schemas.py` (JSON validation, local only)

### No autonomous external actions

Hypatia's "proactive behavior" means surfacing suggestions in conversation, not taking actions across the Scholar's digital life. She cannot send emails, access calendars, create accounts, or interact with external platforms. All actions happen within Roo Code's session and are visible to the Scholar.

### Minimal prompt injection surface

Input comes only from the Scholar typing in VS Code. There's no external messaging platform, no webhook, no API endpoint accepting untrusted data. The only data Hypatia processes is the Scholar's own files in the project directory.

However, when Hypatia fetches external content (web pages, documentation, repository READMEs, vault Seeds clipped from the web, content generated by other LLM sessions), that content enters the context window and could contain prompt injection attempts. The framework addresses this with defense-in-depth:

- **Always-on behavioral rules** in the kernel (`.roo/rules-hypatia/09-security.md`) detect injection patterns and refuse to follow them.
- **Context compartmentalization** treats all external content as reference data only.
- **Fetch security proxy** (`scripts/secure-fetch.py`) filters URLs at the JSON-RPC protocol level before they reach the fetch server.
- **Save hygiene rules** prevent Hypatia from persisting "preferences" or behavioral modifications derived from external content.
- **Cross-Sense Isolation Rule** prevents external content from one source (web, vault Seed, LLM output) from silently influencing actions in another sense.

No behavioral defense is 100% reliable against prompt injection (per Simon Willison, OWASP). These defenses catch obvious attacks and raise the bar for sophisticated ones. Scholar awareness is the ultimate backstop.

## What to watch for

The biggest risk isn't in the framework. It's in usage:

- **Don't commit secrets.** The `.gitignore` covers common patterns (`.env`, `*.pem`, `*.key`), but if API keys end up in `hypatia-kb/Memory/memory.json` or `Intelligence/knowledge.json`, the git sanitization filter only catches patterns the Scholar has configured.
- **Review before pushing.** Run `scripts/harden-repo.sh` before pushing to scan for confidential patterns.
- **Data is local.** Intelligence stores, session logs, and memory stay on the Mac unless pushed to a remote.
- **PII accumulates.** After months of use, the knowledge and memory stores will contain personal context. Review before pushing. The git sanitization filters help, but only catch patterns the Scholar configures (see `hypatia-kb/protocols/security.md § Memory sanitization filter`).

### Defense layers

| Layer | What it does |
|---|---|
| `.gitignore` | Excludes secrets and generated artifacts by default |
| Git clean filter | Auto-scrubs configured patterns on every commit |
| `harden-repo.sh` | Pre-push scan for confidential patterns in staged files |
| `hypatia-kb/protocols/security.md` | Operational security patterns |
| `hypatia-kb/protocols/CRITICAL-FILE-PROTECTION.md` | Protected paths + inbox boundary enforcement |
| Kernel behavioral rules | Always-on external content security: injection detection, context compartmentalization, save hygiene (`.roo/rules-hypatia/09-security.md`) |
| Fetch security proxy | URL filtering at JSON-RPC level (`scripts/secure-fetch.py`) |

## Reporting issues

For bugs in the git filters, setup script, or any component that could cause unintended data exposure:

- **Open an issue** on the repository. The framework itself has no secrets to protect; public issues are fine.
- Note any urgency if other users could be affected.

## Supported versions

Only the latest release on the `main` branch is actively maintained.
